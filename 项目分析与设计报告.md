## 3. 需求分析

### 3.1 产品的功能性需求

#### 3.1.1 整体业务流程图／用例图

系统整体业务流程如下：

1. 用户通过浏览器访问系统前端页面，输入用户名和密码进行登录认证。
2. 系统根据用户角色（管理员/医生）分配不同的功能菜单权限。
3. 管理员可进行全部管理操作，包括医药公司管理、销售地点管理、城市信息管理、药品信息管理、医保政策管理、医药公司政策管理、医生信息管理、必备材料管理。
4. 医生角色可查看药品信息、医保政策、医药公司政策和必备材料信息。
5. 各模块均支持数据的增删改查操作，并通过分页展示数据列表。

**用例图说明：**

- **管理员用例**：登录、医药公司管理（增删改查）、销售地点管理（增删改查、地图定位）、城市信息管理（增删改查）、药品信息管理（增删改查、图片上传、分配销售药店）、医保政策管理（增删改查）、医药公司政策管理（增删改查）、医生信息管理（增删改查）、必备材料管理（增删改查）。
- **医生用例**：登录、查看药品信息、查看医保政策、查看医药公司政策、查看必备材料。

#### 3.1.3 用户登录与权限管理

**1）描述和优先级**

本用例允许用户通过输入用户名和密码登录系统，系统根据用户角色（管理员ROLE_1、医生ROLE_2）动态加载对应的功能菜单权限。管理员拥有全部功能菜单权限，医生拥有药品信息管理、医保政策管理、医药公司政策管理、必备材料管理的查看权限。

> 优先级：高

**2）输入**

用户输入用户名（uname）和密码（pwd）。

**3）操作**

输入用户名和密码后点击"登录"按钮，系统验证用户身份，查询角色权限表获取对应菜单列表。

**4）输出**

登录成功后跳转至系统首页，左侧导航栏显示该角色对应的功能菜单；登录失败则提示错误信息。

#### 3.1.4 医生信息管理

**1）描述和优先级**

本用例允许管理员对医生信息进行增删改查操作，包括医生姓名、年龄、性别、医师级别、联系方式、诊治类别、所属医院等信息的管理。新增医生时系统自动创建对应的登录账号。

> 优先级：高

**2）输入**

用户输入或选择医生姓名、年龄、性别、医师级别（主任医师/普通医师/实习医师）、联系方式、诊治类别（肩部/踝部/膝部/腰部/头部/肘部/腿部）、所属医院。

**3）操作**

- 查询：输入医生姓名关键词点击"搜索"进行模糊查询，或直接浏览分页列表。
- 新增：点击"新增"按钮，填写医生信息后点击"确定"提交。
- 修改：点击对应记录的"编辑"按钮，修改信息后点击"确定"保存。
- 删除：点击对应记录的"删除"按钮，确认后删除该医生信息。

**4）输出**

以表格形式分页显示医生信息列表，包含医生姓名、年龄、性别、医师级别、联系方式、诊治类别、所属医院、创建时间、更新时间等字段。

#### 3.1.5 药品信息管理

**1）描述和优先级**

本用例允许管理员对药品信息进行增删改查操作，包括药品名称、药品成分、药品功能、药品图片等信息的管理，并支持为药品分配销售药店。

> 优先级：高

**2）输入**

用户输入药品名称、药品成分信息、药品功能描述，上传药品图片，选择销售该药品的药店。

**3）操作**

- 查询：输入药品名称关键词点击"搜索"进行模糊查询，或直接浏览分页列表。
- 新增：点击"新增"按钮，填写药品信息并上传图片后点击"确定"提交。
- 修改：点击对应记录的"编辑"按钮，修改信息后点击"确定"保存。
- 删除：点击对应记录的"删除"按钮，确认后删除该药品信息。
- 分配药店：选择药品后，勾选销售该药品的药店进行关联。

**4）输出**

以表格形式分页显示药品信息列表，包含药品名称、药品成分、药品功能、药品图片、发布者、创建时间、更新时间等字段。

### 3.2 产品非功能性需求

#### 3.2.1 用户界面需求

1. 系统采用B/S架构，用户通过浏览器访问，界面风格简洁、美观、易操作。
2. 前端基于Vue.js + Element UI组件库开发，提供统一的表格、表单、弹窗等交互组件。
3. 系统采用左侧导航栏 + 右侧内容区的经典后台管理布局。
4. 所有列表页面支持分页展示，每页默认显示10条数据。
5. 表单输入需进行前端校验，如必填项验证、手机号格式验证等。
6. 操作反馈及时，增删改操作后需给出成功或失败的提示信息。

#### 3.2.2 性能需求

1. 系统页面加载时间不超过3秒。
2. 数据查询响应时间不超过2秒。
3. 系统支持至少50个用户同时在线操作。
4. 数据库查询需建立合理的索引，保证大数据量下的查询效率。

### 3.3 接口需求

#### 3.3.1 外部接口

1. **前端与后端通信接口**：前端Vue.js应用通过HTTP RESTful API与后端Spring Boot服务进行数据交互，数据格式为JSON。
2. **跨域访问接口**：后端通过WebConfig配置类开放CORS跨域访问，允许前端应用从不同端口访问后端API。
3. **Swagger API文档接口**：系统集成Swagger2，提供在线API文档，方便前后端联调。

#### 3.3.2 内部接口

1. **账号认证接口**：提供用户登录认证、密码加密验证功能。
2. **医生信息接口**：提供医生信息的增删改查，关联医师级别和诊治类型。
3. **药品信息接口**：提供药品信息的增删改查，支持图片上传和药店分配。
4. **医药公司接口**：提供医药公司信息的增删改查。
5. **销售地点接口**：提供药店信息的增删改查，包含地理坐标信息。
6. **城市信息接口**：提供城市信息的增删改查，关联行政区域数据。
7. **医保政策接口**：提供医保政策信息的增删改查，关联所属城市。
8. **医药公司政策接口**：提供医药公司政策信息的增删改查。
9. **必备材料接口**：提供就医必备材料信息的增删改查。
10. **权限管理接口**：提供角色权限查询，动态加载菜单。

## 4. 概要设计

### 4.1 系统体系结构

本系统采用前后端分离的B/S（Browser/Server）三层体系结构，分为**表现层**、**业务逻辑层**和**数据访问层**：

```
┌─────────────────────────────────────────────────┐
│                   表现层（前端）                    │
│         Vue.js + Element UI + Axios              │
│         浏览器端渲染，负责用户交互与数据展示           │
├─────────────────────────────────────────────────┤
│              HTTP RESTful API (JSON)              │
├─────────────────────────────────────────────────┤
│                 业务逻辑层（后端）                   │
│    Spring Boot + Spring MVC + Spring Security    │
│    Controller → Service → Mapper 三层架构         │
├─────────────────────────────────────────────────┤
│                   JDBC / MyBatis                  │
├─────────────────────────────────────────────────┤
│                 数据访问层（数据库）                  │
│                  MySQL 8.0                        │
│              数据库：bin_text                       │
└─────────────────────────────────────────────────┘
```

- **表现层**：基于Vue.js框架开发，使用Element UI组件库构建用户界面，通过Axios发送HTTP请求与后端交互。
- **业务逻辑层**：基于Spring Boot框架开发，采用Controller-Service-Mapper三层架构，负责业务逻辑处理、数据校验和权限控制。
- **数据访问层**：使用MySQL 8.0数据库存储系统数据，通过MyBatis框架实现数据持久化操作。

### 7.2 逻辑结构设计

系统数据库实体关系说明如下：

**核心实体及其关系：**

1. **账号（account）** ←1:1→ **医生（doctor）**：每个医生对应一个系统登录账号，通过 `account_id` 关联。
2. **医生（doctor）** ←N:1→ **医师级别（doctor_level）**：每个医生属于一个医师级别（主任医师/普通医师/实习医师），通过 `level_id` 关联。
3. **医生（doctor）** ←N:1→ **诊治类型（treat_type）**：每个医生擅长一种诊治类型（肩部/踝部/膝部等），通过 `type_id` 关联。
4. **药品（drug）** ←M:N→ **销售地点（sale）**：药品与药店为多对多关系，通过中间表 `drug_sale` 关联。
5. **医保政策（medical_policy）** ←N:1→ **城市（city）**：每条医保政策属于一个城市，通过 `city_id` 关联。
6. **角色权限（role_permission）** ←N:1→ **权限菜单（permission）**：通过角色名称与权限ID关联，实现RBAC权限控制。

```
account 1──1 doctor N──1 doctor_level
                   N──1 treat_type

drug M──N sale  (通过 drug_sale 中间表)

medical_policy N──1 city
```

### 7.4 安全性设计

1. **密码安全**：用户密码采用BCrypt加密算法存储，不以明文形式保存在数据库中。
2. **权限控制**：系统基于RBAC（基于角色的访问控制）模型，通过 `role_permission` 表控制不同角色可访问的功能菜单，防止越权操作。
3. **用户名唯一约束**：`account` 表的 `uname` 字段设置唯一索引，防止重复注册。
4. **跨域安全**：后端通过CORS配置控制跨域访问策略。
5. **SQL注入防护**：使用MyBatis参数化查询，避免SQL注入攻击。
6. **数据备份**：定期对MySQL数据库进行备份，防止数据丢失。
